从 JDBC 标准接口 `java.sql.Connection` 看，`PreparedStatement` 相关的标准重载主要就是这 6 个：`prepareStatement(String sql)`、`prepareStatement(String sql, int autoGeneratedKeys)`、`prepareStatement(String sql, int[] columnIndexes)`、`prepareStatement(String sql, String[] columnNames)`、`prepareStatement(String sql, int resultSetType, int resultSetConcurrency)`、`prepareStatement(String sql, int resultSetType, int resultSetConcurrency, int resultSetHoldability)`。也就是说，标准 API 把它们分成了两大组：

一组是“是否/如何返回自动生成键”，另一组是“结果集形态控制”。

标准接口本身并没有提供“同时指定 autoGeneratedKeys 和 resultSetType/resultSetConcurrency/resultSetHoldability”的组合重载。([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/Connection.html?utm_source=chatgpt.com "Connection (Java Platform SE 8 )"))

你可以把这些参数理解成两条不同的配置轴：

第一条轴是 **generated keys 轴**。这里有三种入口：  
`int autoGeneratedKeys`，只能传 `Statement.RETURN_GENERATED_KEYS` 或 `Statement.NO_GENERATED_KEYS`；  
`int[] columnIndexes`，表示返回哪些自动生成列；  
`String[] columnNames`，也是指定返回哪些自动生成列，只是改成按列名指定。标准文档对这几种方法的描述都是“创建一个 capable of returning auto-generated keys 的 PreparedStatement”。([Oracle Docs](https://docs.oracle.com/en/java/javase/21/docs/api/java.sql/java/sql/Statement.html?utm_source=chatgpt.com "Statement (Java SE 21 & JDK 21)"))

第二条轴是 **ResultSet 轴**。这里控制的是查询结果集的行为：  
`resultSetType` 控制游标类型，例如 `TYPE_FORWARD_ONLY`；  
`resultSetConcurrency` 控制是否只读/可更新，例如 `CONCUR_READ_ONLY`、`CONCUR_UPDATABLE`；  
`resultSetHoldability` 控制提交后游标是否保留，例如 `HOLD_CURSORS_OVER_COMMIT`、`CLOSE_CURSORS_AT_COMMIT`。这些值是 `ResultSet` 和 `Statement` 标准的一部分。([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/ResultSet.html?utm_source=chatgpt.com "ResultSet (Java Platform SE 8 )"))

你问的“互斥涉及有哪些标准”，核心答案是：

**1. 互斥首先体现在 API 形状上，而不是文档显式说‘这几个参数互斥’。**  
JDBC 标准没有提供这种签名：  
`prepareStatement(sql, autoGeneratedKeys, resultSetType, resultSetConcurrency, resultSetHoldability)`。  
因此从标准调用层面，**generated-keys 组** 与 **result-set 组** 不能在一次 `prepareStatement(...)` 调用中同时显式指定；你必须二选一进入某个重载。这个“互斥”本质上是 **接口设计上的互斥**。([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/Connection.html?utm_source=chatgpt.com "Connection (Java Platform SE 8 )"))

**2. `autoGeneratedKeys`、`columnIndexes`、`columnNames` 三者也可视为同一层面的互斥选择。**  
它们都是“返回生成键”的不同表达方式：一个是开关，一个是按列下标指定，一个是按列名指定。标准没有要求它们能叠加使用；因为它们本来就是三个不同的重载入口。你在业务代码里通常应当把它们视为“最多选一个”。([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/Connection.html?utm_source=chatgpt.com "Connection (Java Platform SE 8 )"))

**3. `resultSetType` / `resultSetConcurrency` / `resultSetHoldability` 不是彼此完全独立自由组合，是否支持要看驱动。**  
JDBC 标准专门给了 `DatabaseMetaData.supportsResultSetType(type)`、`supportsResultSetConcurrency(type, concurrency)`、`supportsResultSetHoldability(holdability)` 来探测支持性。特别注意，`supportsResultSetConcurrency` 检查的是 **type + concurrency 的组合**，说明标准层面就承认这两者并非任意组合都一定支持。([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/DatabaseMetaData.html?utm_source=chatgpt.com "DatabaseMetaData (Java Platform SE 8 )"))

**4. `holdability` 也不是你传了就一定能用。**  
标准允许驱动只支持某一种 holdability；因此才有 `supportsResultSetHoldability` 和 `Connection.getHoldability()` 这样的接口。很多框架里的逻辑会在没显式设置时退回到连接当前 holdability，这正符合 JDBC 的能力模型。([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/DatabaseMetaData.html?utm_source=chatgpt.com "DatabaseMetaData (Java Platform SE 8 )"))

所以，落到你看到的那段代码，优先级通常会被写成：

先判断是不是要返回 generated keys：  
`autoGeneratedKeys` > `columnIndexes` > `columnNames`

否则再走结果集配置：  
`resultSetType + resultSetConcurrency + resultSetHoldability`

这不是因为 JDBC 文档写了一个“优先级规则”，而是因为 **标准重载本来就分叉了**：一旦你决定走 generated-keys 家族，就没法在同一次标准调用里再显式带上 result-set 家族参数；框架只能自己约定一个分支优先级。([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/Connection.html?utm_source=chatgpt.com "Connection (Java Platform SE 8 )"))

再进一步说，一个比较准确的工程化结论是：

- `getAutoGeneratedKeys`、`getColumnIndexes`、`getColumnNames` 从语义上最好视为“同一组互斥配置”
    
- `resultSetType`、`resultSetConcurrency`、`resultSetHoldability` 是“另一组配置”
    
- 两组之间在 JDBC 标准 `Connection.prepareStatement` 重载层面也是互斥入口
    
- 真正的支持范围还要以驱动的 `DatabaseMetaData` 为准，尤其是 `type + concurrency` 组合支持性 ([Oracle Docs](https://docs.oracle.com/javase/8/docs/api/java/sql/DatabaseMetaData.html?utm_source=chatgpt.com "DatabaseMetaData (Java Platform SE 8 )"))

# generatedKeys 和 resultSet 会严格对应写和读两个分支吗

不严格对应，但**高度倾向于“写 vs 读”的分支划分**——这是一个“工程上成立的二级结论”，但不是 JDBC 规范的硬性约束。

我们把它拆开讲更清楚：

---

## 一、从语义设计上看：确实像“写 vs 读”

### 1. generatedKeys → 偏写路径（INSERT / UPDATE）

- `RETURN_GENERATED_KEYS` 的核心用途是拿 **数据库生成的值**
    
    - 自增主键（MySQL auto_increment）
        
    - sequence（Oracle / PostgreSQL）
        
- 典型调用链：
    
    ```java
    ps.executeUpdate();
    ps.getGeneratedKeys();
    ```
    

👉 这里本质是：  
**执行写操作 → 获取副产物（generated keys）**

---

### 2. resultSet → 偏读路径（SELECT）

- `resultSetType / concurrency / holdability`
    
- 控制的是：
    
    - 游标滚动（forward / scroll）
        
    - 是否可更新
        
    - commit 后是否关闭
        

典型调用：

```java
ResultSet rs = ps.executeQuery();
```

👉 本质是：  
**执行读操作 → 获取查询结果集**

---

## 二、但规范上：它们不是严格对立的两类

JDBC 设计里有几个关键点，打破“绝对读写分离”的直觉：

---

### 1. `PreparedStatement` 本身是统一模型

同一个 `PreparedStatement` 可以：

- `executeQuery()` → 返回 ResultSet
    
- `executeUpdate()` → 返回 update count
    
- `execute()` → 混合（甚至多结果）
    

👉 说明 JDBC **没有强制把 Statement 分成“读/写两种类型”**

---

### 2. 写操作也可以“返回 ResultSet”

例如：

#### MySQL

```sql
INSERT ... RETURNING id
```

#### PostgreSQL（更典型）

```sql
INSERT INTO t (...) VALUES (...) RETURNING id;
```

👉 这里你会用：

```java
ps.executeQuery();  // 直接拿 ResultSet
```

而不是 `getGeneratedKeys()`

---

### 3. 查询也可能触发 generated keys（极少见但存在）

例如：

- 某些 driver/DB 在 trigger 中产生值
    
- 或 executeBatch + mixed statement
    

👉 规范允许 `getGeneratedKeys()` 在 execute 后调用，而不是限定必须是 INSERT

---

## 三、为什么你看到的代码“强分支”

你那段代码：

```java
if (autoGeneratedKeys) {
   ...
} else if (columnIndexes) {
   ...
} else if (columnNames) {
   ...
} else {
   // resultSetType + concurrency + holdability
}
```

本质不是 JDBC 要求，而是：

### 👉 框架在做“意图推断”

可以理解为：

|条件|推断意图|
|---|---|
|generatedKeys|我要 INSERT 并拿 key|
|columnIndexes/names|我要精确拿 key|
|否则|我要控制 ResultSet 行为（SELECT 或 cursor）|

---

## 四、真正的底层原因（关键 insight）

### 👉 不是“读写互斥”，而是“两个 orthogonal 能力无法在 API 中组合表达”

JDBC 的问题是：

> ❗ 没有提供一个统一构造器同时表达：
> 
> - 我要 generated keys
>     
> - 还要 scrollable result set
>     

所以：

- 不是语义冲突
    
- 是 **API 表达能力限制**
    

---

## 五、可以这样建立一个更准确的认知

比“读 vs 写”更精确的是：

### 👉 两个维度：

#### 维度 1：执行结果类型

- update count
    
- result set
    
- generated keys（副产物）
    

#### 维度 2：结果行为控制

- resultSetType
    
- concurrency
    
- holdability
    

---

而 JDBC 的问题是：

> ❗ **generatedKeys 和 resultSet 行为控制分布在不同构造函数族，无法组合**

---

## 六、工程上可以用的结论（很重要）

你可以记住这几个“近似成立”的规则：

### ✅ 1. generatedKeys ≈ 写路径（但不是绝对）

### ✅ 2. resultSetType 等 ≈ 读路径（但 UPDATE 也可能用到可更新 ResultSet）

### ❗ 3. 两者不能同时指定，是 API 限制，不是语义冲突

### ❗ 4. 真正决定行为的是：

- SQL 类型（SELECT / INSERT）
    
- driver 能力
    
- executeXXX 方法
    

---

## 七、结合你在做中间件的视角（更关键）

在 DongDAL / JDBC 中间件设计里：

你其实是在做：

### 👉 “Statement capability routing”

你必须提前决定：

- 我要不要 generated keys
    
- 我要不要 scrollable cursor
    

因为：

> ❗ 一旦选了一个 prepareStatement 重载，另一个能力就没法表达了

这也是为什么必须写成 if-else，而不是 merge。

---

## 最后一条总结（最本质）

👉 **generatedKeys 和 resultSet 不是“写 vs 读”的严格分类，而是 JDBC API 在“结果表达能力”上的两个分叉设计；工程上常映射为写/读，但本质是接口建模限制，而不是语义对立。**
