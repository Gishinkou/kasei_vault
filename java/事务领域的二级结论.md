很好，这个问题本质是在建立**“事务直觉模型（mental model）”**。  
我给你一组在工程上**非常有用的“二级结论”**（不是绝对真理，但在绝大多数场景成立），并配上“什么时候会失效”。

你可以把它们当成：**快速判断系统是否正确的 heuristics（经验定律）**。

---

# 一、关于“事务与载体”的直觉

## 1️⃣ 👉 事务 ≈ Connection 生命周期的一段区间（最核心）

```text
begin (autoCommit=false) → execute → commit/rollback
```

✅ 用途：

- 判断事务是否断裂
    
- 判断中间件是否正确
    

❌ 失效场景：

- XA / 分布式事务（一个逻辑事务对应多个 connection）
    

---

## 2️⃣ 👉 同一个事务 = 同一个 Connection

这是你最重要的判断规则。

✅ 如果你看到：

```text
一个事务中使用了多个 connection
```

👉 直接判定：**不是同一个本地事务**

❌ 失效场景：

- 分布式事务（XA / TCC / Saga）
    

---

## 3️⃣ 👉 Connection 是“事务的唯一载体”，不是线程

👉 线程只是 Spring 的实现手段

这条可以帮你避免很多误判，比如：

```text
换线程 ≠ 一定丢事务
换 connection = 一定丢事务
```

---

# 二、关于“线程模型”的直觉

## 4️⃣ 👉 通常成立：一个事务只在一个线程内

（你提到的这一条）

成立原因：

- Spring 用 ThreadLocal 绑定 connection
    

---

### 你可以这样用这条规则：

👉 看到：

```java
@Transactional
+ new Thread / @Async
```

👉 直接警觉：

```text
事务很可能丢失 ❗
```

---

❌ 失效场景：

- Reactive（R2DBC）
    
- 手动传递 connection（极少见）
    

---

## 5️⃣ 👉 同一个线程 ≠ 一定同一个事务

这是很多人忽略的反例。

例如：

```java
foo(); // 事务1
bar(); // 事务2
```

👉 同一个线程，但：

```text
两个独立事务
```

---

所以你要建立：

```text
线程是“可能关联事务”，但不是“事务边界”
```

---

# 三、关于“事务边界”的直觉

## 6️⃣ 👉 事务边界通常等于方法边界（在 Spring 中）

```java
@Transactional
public void foo() { ... }
```

👉 默认：

```text
方法开始 → 事务开始
方法结束 → commit/rollback
```

---

❌ 失效场景：

- 内部调用（self-invocation）
    
- 编程式事务
    
- propagation 改变
    

---

## 7️⃣ 👉 嵌套调用默认不会创建新事务

```java
@Transactional
public void A() {
    B();
}

@Transactional
public void B() {}
```

👉 默认：

```text
A 和 B 是同一个事务
```

（Propagation.REQUIRED）

---

❌ 失效场景：

```java
REQUIRES_NEW
```

---

## 8️⃣ 👉 REQUIRES_NEW = 一定换 Connection

这是一个非常实用的工程直觉：

```java
@Transactional(propagation = REQUIRES_NEW)
```

👉 本质：

```text
挂起当前 connection
新拿一个 connection
```

---

👉 所以你可以推导：

```text
REQUIRES_NEW ≈ 新事务 ≈ 新 connection
```

---

# 四、关于“连接池 + 事务”的直觉

## 9️⃣ 👉 Connection.close() 通常不关闭连接

在连接池中：

```text
close = 归还连接池
```

---

👉 推论：

```text
连接会被复用 → 状态必须被清理
```

---

## 🔟 👉 连接复用 = 状态泄露风险

你可以形成一个强直觉：

```text
只要用了连接池 → 一定存在状态残留风险
```

---

👉 所以：

```text
中间件的本质工作之一 = 清理 connection 状态
```

---

# 五、关于“读写一致性”的直觉

## 1️⃣1️⃣ 👉 同一个事务内一定“读己之写”

```java
insert();
select();
```

👉 一定能读到刚插入的数据（同 connection）

---

❌ 失效场景：

- 读写分离 + 路由错误
    
- 中间件错误切换 connection
    

---

👉 所以：

```text
事务内读不到写 → 99% 是 connection 被切了
```

（这个非常有用）

---

## 1️⃣2️⃣ 👉 事务外读写不保证一致

```text
事务1写
事务2读
```

👉 是否能读到：

- 取决于隔离级别
    
- 取决于是否走主库
    

---

# 六、关于“中间件设计”的直觉（你重点关注）

## 1️⃣3️⃣ 👉 事务期间不能切库/切 connection

👉 否则：

```text
事务语义直接失效
```

---

👉 所以：

```text
路由必须“事务感知”
```

---

## 1️⃣4️⃣ 👉 readOnly 往往参与路由决策

👉 常见规则：

```text
readOnly=true  → 从库
readOnly=false → 主库
```

---

👉 推论：

```text
readOnly 状态残留 = 路由错误
```

---

## 1️⃣5️⃣ 👉 autoCommit=false 时必须固定路由

因为：

```text
事务已经开始 → 必须固定到一个 connection / 一个库
```

---

👉 所以很多中间件会：

```text
if (inTransaction) → 禁止切换数据源
```

---

# 七、关于“异常与事务”的直觉

## 1️⃣6️⃣ 👉 默认：RuntimeException 才会触发回滚

Spring 默认：

```text
RuntimeException / Error → rollback
Checked Exception → commit
```

---

👉 很多人误以为：

```text
任何异常都会回滚 ❌
```

---

## 1️⃣7️⃣ 👉 rollback 不等于释放资源

```text
rollback → 结束事务
但 connection 还在
```

---

👉 所以仍然需要：

```text
close → 归还连接池
```

---

# 八、关于“危险信号”的直觉（非常实用）

看到这些代码/现象，你可以立刻警觉：

---

## 🚨 1️⃣ @Transactional + 异步

```java
@Async
@Transactional
```

👉 基本是错的（事务不会传播）

---

## 🚨 2️⃣ 事务中出现多 DataSource

👉 高概率：

```text
事务断裂
```

---

## 🚨 3️⃣ 事务中读不到刚写的数据

👉 优先怀疑：

```text
connection 被切换了
```

---

## 🚨 4️⃣ 慢 SQL + 锁等待异常

👉 优先怀疑：

```text
事务未提交 / autoCommit=false 残留
```

---

# 九、把这些收敛成一组“核心心法”

你可以记这 5 条就够了：

---

## 🧠 心法 1

```text
事务 = Connection 的状态区间
```

---

## 🧠 心法 2

```text
同一事务 = 同一 Connection
```

---

## 🧠 心法 3

```text
Spring 只是用线程帮你“固定住 Connection”
```

---

## 🧠 心法 4

```text
连接池 = 状态复用风险
```

---

## 🧠 心法 5

```text
一切事务问题，本质都是：
Connection 用错了 / 换了 / 没清理
```
